# 7. Revisão técnica

## 7.1 SOAP e REST

### O que é SOAP?

SOAP significa **Simple Object Access Protocol**.

É um protocolo usado para troca estruturada de mensagens entre aplicações.

Características principais:

- utiliza normalmente XML;
- define uma estrutura rígida para as mensagens;
- pode utilizar um contrato WSDL;
- é frequente em sistemas empresariais, bancos, seguradoras e sistemas antigos;
- possui mecanismos normalizados para erros, segurança e transações;
- tende a ser mais pesado e formal do que REST.

### O que é WSDL?

WSDL significa **Web Services Description Language**.

É um ficheiro XML que descreve:

- operações disponíveis;
- parâmetros de entrada;
- tipos de dados;
- respostas;
- endereço do serviço;
- formato das mensagens.

Pode ser visto como um contrato formal entre quem fornece e quem consome o serviço.

### O que é REST?

REST significa **Representational State Transfer**.

Não é um protocolo rígido como SOAP. É um estilo arquitetural frequentemente usado sobre HTTP.

Características principais:

- usa recursos identificados por URLs;
- utiliza métodos HTTP;
- normalmente troca dados em JSON;
- costuma ser mais simples e leve;
- é muito usado em aplicações web e APIs modernas.

### Métodos HTTP principais

| Método | Utilização habitual |
|---|---|
| `GET` | Obter dados |
| `POST` | Criar um recurso ou executar uma operação |
| `PUT` | Substituir ou atualizar integralmente um recurso |
| `PATCH` | Atualizar parcialmente um recurso |
| `DELETE` | Eliminar um recurso |

### Códigos HTTP importantes

| Código | Significado |
|---|---|
| `200` | Pedido executado com sucesso |
| `201` | Recurso criado |
| `204` | Sucesso sem conteúdo na resposta |
| `400` | Pedido inválido |
| `401` | Não autenticado |
| `403` | Sem autorização |
| `404` | Recurso não encontrado |
| `409` | Conflito |
| `500` | Erro interno do servidor |
| `503` | Serviço temporariamente indisponível |

### SOAP versus REST

| SOAP | REST |
|---|---|
| Protocolo formal | Estilo arquitetural |
| Normalmente XML | Normalmente JSON |
| Contrato WSDL frequente | Documentação OpenAPI frequente |
| Mais rígido | Mais flexível |
| Mais pesado | Mais leve |
| Muito presente em sistemas empresariais | Muito presente em APIs modernas |

### Resposta curta para entrevista

> SOAP é um protocolo formal, normalmente baseado em XML e frequentemente descrito através de WSDL. REST é um estilo arquitetural geralmente usado sobre HTTP, com recursos, métodos como GET e POST e respostas frequentemente em JSON. SOAP é comum em integrações empresariais mais rígidas; REST costuma ser mais simples e leve.

## 7.2 Como investigar um erro num webservice

Um erro num webservice pode estar em várias camadas:

- pedido enviado pelo cliente;
- autenticação ou autorização;
- rede;
- aplicação;
- base de dados;
- serviço externo;
- transformação dos dados;
- configuração;
- versão do software.

### Método de diagnóstico

#### 1. Perceber o comportamento esperado

Antes de alterar código, devo perceber:

- o que deveria acontecer;
- o que aconteceu realmente;
- em que ambiente ocorreu;
- desde quando ocorre;
- se acontece sempre ou apenas em algumas situações.

#### 2. Reproduzir o problema

Tentar reproduzir com:

- os mesmos dados;
- o mesmo utilizador;
- o mesmo ambiente;
- a mesma sequência de operações.

Um erro reproduzível é muito mais fácil de investigar.

#### 3. Consultar logs

Procurar:

- data e hora;
- identificador do pedido;
- mensagem de erro;
- stack trace;
- parâmetros relevantes;
- código HTTP;
- duração do pedido;
- serviço ou método chamado.

#### 4. Validar o pedido

Confirmar:

- URL correta;
- método HTTP correto;
- cabeçalhos;
- token de autenticação;
- formato dos dados;
- campos obrigatórios;
- encoding;
- tipos de dados.

#### 5. Verificar a resposta

Analisar:

- código HTTP;
- corpo da resposta;
- mensagem de erro;
- tempo de resposta;
- cabeçalhos;
- dados incompletos ou inesperados.

#### 6. Isolar a camada do erro

Perguntar:

- o pedido chegou à aplicação?
- a aplicação conseguiu aceder à base de dados?
- a query devolveu os dados esperados?
- o serviço externo respondeu?
- ocorreu uma exceção?
- houve timeout?
- houve uma alteração recente?

#### 7. Corrigir e testar

Depois da correção:

- testar o caso que falhava;
- testar casos normais;
- testar casos limite;
- confirmar que não foram introduzidas regressões;
- documentar a causa e a solução.

### Resposta curta para entrevista

> Primeiro procuro perceber e reproduzir o problema. Depois analiso logs, pedido, resposta e alterações recentes. Tento isolar se o erro está no cliente, na aplicação, na base de dados ou num serviço externo. Só depois altero o código. Por fim, testo a correção e verifico possíveis regressões.

## 7.3 SQL

SQL é usado para consultar, inserir, alterar e eliminar dados numa base de dados relacional.

### `INNER JOIN`

Devolve apenas registos com correspondência nas duas tabelas.

```sql
SELECT c.nome, e.numero_encomenda
FROM clientes AS c
INNER JOIN encomendas AS e
    ON e.cliente_id = c.id;
```

Se um cliente não tiver encomendas, não aparece no resultado.

### `LEFT JOIN`

Devolve todos os registos da tabela da esquerda, mesmo quando não existe correspondência na tabela da direita.

```sql
SELECT c.nome, e.numero_encomenda
FROM clientes AS c
LEFT JOIN encomendas AS e
    ON e.cliente_id = c.id;
```

Permite encontrar, por exemplo, clientes sem encomendas.

```sql
SELECT c.nome
FROM clientes AS c
LEFT JOIN encomendas AS e
    ON e.cliente_id = c.id
WHERE e.id IS NULL;
```

### Stored procedure

Uma stored procedure é um conjunto de instruções SQL guardado na base de dados.

Pode:

- receber parâmetros;
- executar várias queries;
- aplicar regras;
- controlar transações;
- devolver resultados;
- centralizar lógica de acesso aos dados.

Exemplo conceptual:

```sql
CREATE PROCEDURE ObterEncomendasCliente
    @ClienteId INT
AS
BEGIN
    SELECT *
    FROM encomendas
    WHERE cliente_id = @ClienteId;
END;
```

### Transação

Uma transação agrupa várias operações que devem funcionar como uma unidade.

Propriedades ACID:

- **Atomicidade:** todas as operações são concluídas ou nenhuma é;
- **Consistência:** os dados mantêm as regras definidas;
- **Isolamento:** transações concorrentes não interferem indevidamente;
- **Durabilidade:** depois do `COMMIT`, os dados permanecem guardados.

Exemplo:

```sql
BEGIN TRANSACTION;

UPDATE contas
SET saldo = saldo - 100
WHERE id = 1;

UPDATE contas
SET saldo = saldo + 100
WHERE id = 2;

COMMIT;
```

Se ocorrer um erro, deve ser feito `ROLLBACK`.

### Índice

Um índice acelera a procura de dados, de forma semelhante ao índice de um livro.

Pode melhorar:

- pesquisas;
- filtros;
- ordenações;
- `JOIN`;
- acesso por chave.

Mas também tem custos:

- ocupa espaço;
- torna inserções e atualizações mais lentas;
- necessita de manutenção.

Não se deve criar índices indiscriminadamente.

### `DELETE`, `TRUNCATE` e `DROP`

| Comando | Efeito |
|---|---|
| `DELETE` | Elimina linhas, podendo usar `WHERE` |
| `TRUNCATE` | Elimina todas as linhas de forma mais direta |
| `DROP` | Elimina a própria tabela e a sua estrutura |

Exemplos:

```sql
DELETE FROM utilizadores
WHERE id = 10;
```

```sql
TRUNCATE TABLE logs_temporarios;
```

```sql
DROP TABLE tabela_antiga;
```

### Como investigar uma query lenta

1. confirmar o tempo de execução;
2. analisar o plano de execução;
3. verificar índices;
4. procurar leituras excessivas;
5. evitar `SELECT *` quando não é necessário;
6. verificar filtros e `JOIN`;
7. analisar o volume de dados;
8. verificar bloqueios;
9. validar se funções impedem o uso de índices;
10. testar a alteração com dados realistas.

### Resposta curta para entrevista

> Para analisar uma query lenta, começo pelo plano de execução. Verifico índices, filtros, joins, leituras e volume de dados. Confirmo também se existem bloqueios ou funções que impeçam a utilização dos índices. Depois comparo o desempenho antes e depois da alteração.

## 7.4 Fundamentos de C

### Variável por valor

Quando uma função recebe um valor, recebe normalmente uma cópia.

```c
void incrementar(int valor) {
    valor++;
}
```

A variável original não é alterada.

### Ponteiro

Um ponteiro guarda o endereço de memória de outra variável.

```c
void incrementar(int *valor) {
    (*valor)++;
}
```

Chamada:

```c
int numero = 5;
incrementar(&numero);
```

Depois da função, `numero` passa a valer `6`.

### `struct`

Uma `struct` agrupa dados relacionados.

```c
typedef struct {
    int id;
    char nome[50];
    double saldo;
} Cliente;
```

Permite representar uma entidade com vários campos.

### Stack e heap

#### Stack

- memória automática;
- usada para variáveis locais;
- muito rápida;
- libertada automaticamente ao terminar a função;
- tamanho limitado.

#### Heap

- memória dinâmica;
- obtida com `malloc`, `calloc` ou `realloc`;
- deve ser libertada com `free`;
- permite estruturas cujo tamanho só é conhecido em execução;
- erros podem causar fugas de memória e corrupção.

### `malloc` e `free`

```c
int *numeros = malloc(10 * sizeof(int));

if (numeros == NULL) {
    /* Falha de alocação */
    return 1;
}

/* Utilização de numeros */

free(numeros);
numeros = NULL;
```

Boas práticas:

- verificar se `malloc` devolveu `NULL`;
- libertar a memória quando deixa de ser necessária;
- evitar usar memória depois de `free`;
- colocar o ponteiro a `NULL` quando adequado;
- garantir que cada alocação tem uma libertação correspondente.

### Memory leak

Ocorre quando é alocada memória que deixa de estar acessível sem ter sido libertada.

Consequências:

- consumo crescente de memória;
- degradação de desempenho;
- possível falha da aplicação.

### Segmentation fault

Ocorre quando o programa tenta aceder a memória inválida.

Causas comuns:

- ponteiro `NULL`;
- ponteiro não inicializado;
- acesso fora dos limites de um array;
- uso de memória depois de `free`;
- escrita numa área de memória protegida.

### Exemplo de erro

```c
int *p = NULL;
*p = 10;
```

Este código tenta escrever através de um ponteiro nulo.

### Resposta curta para entrevista

> Um ponteiro guarda um endereço de memória. Em C é importante validar ponteiros, respeitar os limites dos arrays e gerir corretamente a memória dinâmica. Uma fuga de memória ocorre quando a memória alocada não é libertada. Um segmentation fault ocorre normalmente devido a um acesso inválido à memória.

## 7.5 Debugging e análise de logs

Debugging é o processo de localizar, compreender e corrigir defeitos.

### Processo recomendado

1. reproduzir o problema;
2. reduzir o problema ao caso mínimo;
3. recolher evidências;
4. analisar logs;
5. usar breakpoints;
6. inspecionar variáveis;
7. verificar fluxo de execução;
8. formular uma hipótese;
9. testar a hipótese;
10. corrigir e criar um teste de regressão.

### O que um bom log deve permitir perceber

- quando ocorreu;
- em que componente;
- qual foi o pedido;
- qual o utilizador ou processo;
- qual o identificador de correlação;
- qual foi o erro;
- qual a duração;
- qual o estado relevante.

### Evitar nos logs

- passwords;
- tokens;
- dados pessoais desnecessários;
- informação bancária;
- grandes quantidades de dados sem utilidade;
- mensagens vagas como “erro desconhecido”.

### Níveis de log

| Nível | Utilização |
|---|---|
| `DEBUG` | Informação detalhada para diagnóstico |
| `INFO` | Eventos normais importantes |
| `WARN` | Situação anormal, mas recuperável |
| `ERROR` | Operação que falhou |
| `FATAL` | Falha crítica da aplicação |

### Resposta curta para entrevista

> Procuro primeiro reproduzir o problema e recolher evidências. Uso logs, breakpoints e inspeção de variáveis para localizar a origem. Trabalho com hipóteses verificáveis e, depois da correção, adiciono testes ou validações para evitar regressões.

## 7.6 Git

Git é um sistema de controlo de versões distribuído.

Permite:

- guardar o histórico;
- trabalhar em branches;
- colaborar com outros programadores;
- rever alterações;
- recuperar versões;
- integrar trabalho através de merges ou pull requests.

### Conceitos principais

#### Repository

Conjunto de ficheiros e respetivo histórico.

#### Commit

Registo de um conjunto coerente de alterações.

Um bom commit deve:

- ter um objetivo claro;
- ser relativamente pequeno;
- ter uma mensagem compreensível;
- não misturar assuntos diferentes.

#### Branch

Linha de desenvolvimento independente.

Exemplo:

```bash
git checkout -b fix/correcao-timeout
```

#### Merge

Integra as alterações de uma branch noutra.

#### Pull request

Pedido para rever e integrar alterações.

Ajuda a:

- validar código;
- discutir decisões;
- executar testes automáticos;
- garantir qualidade.

### Conflito

Um conflito ocorre quando o Git não consegue decidir automaticamente como combinar alterações.

Deve ser resolvido analisando o código, não escolhendo cegamente uma versão.

### Resposta curta para entrevista

> Uso Git para controlar versões, criar branches, fazer commits pequenos e integrar alterações através de merge ou pull requests. Em caso de conflito, analiso a intenção das duas alterações, resolvo manualmente e volto a executar os testes.

## 7.7 Jenkins

Jenkins é uma ferramenta de automação muito usada em integração e entrega contínuas.

Pode executar automaticamente:

- compilação;
- testes;
- análise estática;
- criação de artefactos;
- deploy;
- notificações.

### Pipeline típico

1. obter o código;
2. instalar dependências;
3. compilar;
4. executar testes;
5. analisar qualidade;
6. gerar pacote;
7. publicar;
8. fazer deploy, quando aprovado.

### CI e CD

#### Continuous Integration

Os programadores integram alterações frequentemente e o sistema valida automaticamente:

- compilação;
- testes;
- qualidade;
- compatibilidade.

#### Continuous Delivery ou Deployment

Automatiza a preparação ou publicação de versões.

### Quando uma pipeline falha

Verificar:

- etapa que falhou;
- logs;
- alteração recente;
- dependências;
- variáveis de ambiente;
- credenciais;
- testes;
- espaço em disco;
- rede;
- diferenças entre ambientes.

### Resposta curta para entrevista

> Jenkins permite automatizar pipelines de compilação, testes e entrega. Quando uma pipeline falha, começo pela etapa exata e pelos logs, verifico alterações recentes, dependências e diferenças de configuração entre ambientes.

## 7.8 Jira

Jira é uma ferramenta de gestão de trabalho e acompanhamento de tarefas.

Pode organizar:

- bugs;
- histórias de utilizador;
- tarefas;
- prioridades;
- responsáveis;
- estados;
- versões;
- sprints.

### Uma boa tarefa deve conter

- descrição clara;
- contexto;
- comportamento esperado;
- comportamento observado;
- critérios de aceitação;
- passos para reproduzir;
- prioridade;
- anexos ou logs relevantes.

### Exemplo de fluxo

```text
To Do → In Progress → Code Review → Testing → Done
```

### Resposta curta para entrevista

> Uso Jira para acompanhar tarefas, bugs e prioridades. Procuro manter cada ticket atualizado, com descrição clara, critérios de aceitação, evidências e estado real do trabalho.

## 7.9 Manutenção de aplicações existentes

Manutenção não significa apenas corrigir bugs.

Pode incluir:

- correções;
- pequenas evoluções;
- atualização de dependências;
- otimização;
- segurança;
- melhoria de logs;
- adaptação a novos sistemas;
- suporte a utilizadores;
- análise de incidentes.

### Antes de alterar uma aplicação antiga

1. perceber o objetivo da aplicação;
2. identificar componentes;
3. consultar documentação;
4. analisar logs;
5. localizar testes;
6. executar a aplicação;
7. perceber dependências;
8. avaliar impacto;
9. fazer uma alteração pequena;
10. validar regressões.

### Regra importante

> Numa aplicação crítica, uma alteração pequena, bem compreendida e testada é geralmente preferível a uma grande reescrita sem conhecimento completo do sistema.
